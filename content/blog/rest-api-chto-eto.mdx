Слово «API» звучит пугающе, но за ним стоит очень простая идея. Бизнес использует API каждый день — иногда не осознавая этого. Давайте разберёмся, что это такое, почему это важно и как это работает без погружения в код.

## Что такое API: аналогия для не-программистов

Представьте ресторан. Вы — клиент, кухня — сервер (место, где «обрабатывается» ваш запрос). Между вами — официант.

Вы не идёте на кухню сами. Вы не кричите повару напрямую. Вы делаете заказ официанту, официант передаёт его на кухню, кухня готовит, официант приносит результат.

**API — это официант.** Это стандартный интерфейс, через который одна программа «общается» с другой.

Когда вы оплачиваете покупку картой — ваш банк и терминал магазина говорят друг с другом через API платёжной системы. Когда Яндекс.Карты показывают пробки — они получают данные через API Яндекса. Когда CRM отправляет SMS-уведомление клиенту — это API SMS-сервиса.

## REST: что означает аббревиатура

**REST** (Representational State Transfer) — это архитектурный стиль для создания API. Не технология, а набор правил и ограничений.

Если API — это «официант», то REST — это стандарт обслуживания: как официант принимает заказ, в каком формате отдаёт блюда, как ведёт себя при ошибках.

REST-API работает через HTTP — тот же протокол, что и обычные веб-сайты. Это делает его универсальным: любое приложение, умеющее делать HTTP-запросы (а это все современные языки программирования), может использовать REST API.

### Основные методы (команды) REST

| Метод | Что делает | Аналогия |
|-------|-----------|----------|
| GET | Получить данные | «Дайте мне список заказов» |
| POST | Создать новый объект | «Создайте новый заказ» |
| PUT | Обновить существующий объект | «Измените статус заказа» |
| DELETE | Удалить объект | «Отмените этот заказ» |

## Примеры использования API в бизнесе

Это не теория — вот конкретные сценарии, с которыми я работаю как разработчик:

### 1. Интеграция CRM с Telegram-ботом

Клиент оставляет заявку в [Telegram-боте](/uslugi/telegram-bot/) → бот через API создаёт контакт в AmoCRM → менеджер получает уведомление в CRM и в Telegram → все данные синхронизированы автоматически.

Без API: менеджер вручную копирует данные из Telegram в CRM. При 20+ заявках в день — это 30–60 минут ежедневной рутины плюс неизбежные ошибки.

### 2. Обновление цен на маркетплейсах

Интернет-магазин продаёт на Wildberries и Ozon. Цены нужно менять синхронно. Через [API Wildberries и Ozon](/blog/integraciya-wildberries-ozon-api/) один запрос обновляет цены на обеих платформах одновременно.

### 3. Платёжная система на сайте

Клиент нажимает «Оплатить» → ваш сайт через API ЮКассы или Сбербанка создаёт платёж → клиент проходит оплату → API возвращает подтверждение → ваш сайт обновляет статус заказа.

### 4. СМС и email-уведомления

Заказ отправлен → API СМС-сервиса (СМСПИЛОТ, TextBack) → клиент получает уведомление → никакого ручного труда.

### 5. Синхронизация с 1С

1С-предприятие хранит данные о товарах, заказах, клиентах. API-интеграция позволяет сайту в реальном времени получать актуальные остатки, цены, данные клиентов — без ручного экспорта/импорта файлов.

## REST vs GraphQL vs gRPC: когда что выбирать

Разработчики иногда предлагают разные архитектуры. Вот когда что используется:

### REST API
**Когда:** стандартные веб-приложения и мобильные приложения, публичные API, интеграции между сервисами.

**Плюсы:** простота, широкая поддержка, кешируемость, легко тестировать в браузере.

**Используйте REST если:** вам нужна стандартная интеграция с CRM, платёжными системами, маркетплейсами, Telegram API.

### GraphQL
**Когда:** сложные приложения, где клиент сам выбирает, какие данные получить.

**Плюсы:** гибкость запросов, меньше лишних данных в ответе.

**Используйте GraphQL если:** у вас сложный фронтенд, который делает много разнообразных запросов, или мобильное приложение с разными экранами и разными наборами данных.

### gRPC
**Когда:** микросервисная архитектура, высоконагруженные системы.

**Плюсы:** очень быстрый, строгая типизация, автогенерация кода.

**Используйте gRPC если:** у вас много взаимодействующих сервисов на бэкенде с требованиями к производительности.

**Для большинства проектов малого и среднего бизнеса — REST API.** Это проверенный стандарт с огромной документацией и поддержкой.

## Как устроен REST API: технические детали (без паники)

Если вы хотите говорить с разработчиком на одном языке — вот минимум знаний.

### Endpoint (эндпоинт)
URL, по которому доступен конкретный ресурс. Например:
```
GET https://api.example.com/orders        — список заказов
GET https://api.example.com/orders/123   — конкретный заказ №123
POST https://api.example.com/orders      — создать новый заказ
```

### Формат данных: JSON
Большинство REST API обмениваются данными в формате JSON — это текст, структурированный как пары «ключ: значение»:
```json
{
  "order_id": 123,
  "client": "Иванов Иван",
  "status": "shipped",
  "total": 4500
}
```

### HTTP-коды ответа
API отвечает кодами, которые сигнализируют о результате:
- **200** — успешно
- **201** — создано
- **400** — ошибка в запросе (вы что-то не так передали)
- **401** — не авторизованы
- **404** — объект не найден
- **500** — ошибка сервера

### Аутентификация
Чтобы API не мог использовать кто попало, используется аутентификация — API Key (секретный ключ) или OAuth токен. Это как пароль для вашего API.

## Как заказать разработку API

Если вам нужна собственная API-интеграция, вот что важно подготовить:

**1. Опишите задачу, а не техническое решение.** «Хочу, чтобы заявки из сайта автоматически попадали в CRM» — это правильная постановка. «Нужен REST API с методом POST для создания контактов» — это то, что скажет разработчик, не вы.

**2. Перечислите системы, которые нужно интегрировать.** CRM (AmoCRM, Битрикс24), мессенджеры (Telegram, WhatsApp), платёжные системы, маркетплейсы, 1С — список систем определяет сложность.

**3. Спросите про документацию.** Хороший разработчик пишет документацию к API — список эндпоинтов, форматы запросов и ответов, примеры. Без документации вы будете зависеть от разработчика при каждом изменении.

**4. Уточните обработку ошибок.** Что происходит, если интегрируемый сервис недоступен? API должен корректно обрабатывать ошибки, а не падать.

**5. Обсудите безопасность.** API с доступом к клиентским данным должен быть защищён: HTTPS, аутентификация, ограничение по IP при необходимости.

## Стоимость разработки API-интеграции

Стоимость зависит от сложности:

| Тип задачи | Примерный бюджет | Сроки |
|-----------|-----------------|-------|
| Простая интеграция (1 система → 1 система) | 15 000–40 000 ₽ | 1–2 недели |
| Несколько систем, двусторонняя синхронизация | 40 000–100 000 ₽ | 3–5 недель |
| Кастомное API для собственного продукта | от 100 000 ₽ | 6–12 недель |

Простые интеграции (CRM + Telegram, сайт + платёжная система) — это стандартные задачи, которые я делаю регулярно. Они предсказуемые по срокам и стоимости.

## Итог

API — это стандартный способ «соединить» разные программы между собой. REST API — наиболее популярный стандарт для веб-приложений. Бизнес использует API для автоматизации рутины: обработки заявок, синхронизации данных, уведомлений клиентов.

Если у вас есть задача «сделать так, чтобы данные из А автоматически попадали в Б» — это почти наверняка решается через API. Объясните задачу — разработчик предложит решение.
