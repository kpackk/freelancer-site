# Разработка API интеграций: как соединить ваши сервисы без головной боли

Магазин принимает заказ на сайте — менеджер вручную переносит его в 1С. Клиент написал в Telegram — менеджер идёт в CRM и создаёт сделку. Курьер принял посылку — оператор заходит на сайт доставки и вносит трекинг-номер.

Всё это одни и те же ручные действия, которые повторяются десятки раз в день. И всё это можно убрать — через API интеграцию.

В этой статье расскажу, что такое API интеграция на языке бизнеса, какие задачи решает, сколько стоит, и покажу реальный кейс из практики.

---

## Что такое API интеграция — объясняем бизнесу

### API простым языком: «переводчик» между программами

API (Application Programming Interface) — это набор правил, по которым одна программа умеет говорить с другой. Представьте, что у вас два сотрудника: один говорит по-русски, второй — только по-английски. Чтобы они работали вместе, нужен переводчик.

API — это и есть переводчик между программами. Ваш сайт не понимает язык 1С, CRM не понимает язык службы доставки. API интеграция выступает посредником: принимает данные от одной системы, переводит их в нужный формат и передаёт другой.

Когда вы оплачиваете покупку картой на сайте — это API интеграция между магазином и банком. Когда трекинг-номер автоматически появляется в заказе — это API интеграция с курьерской службой.

### Примеры: 1С + сайт, Telegram + CRM, магазин + доставка

Вот типичные интеграции из реальной практики:

- **Сайт интернет-магазина + 1С** — заказы автоматически уходят в 1С, остатки товаров на сайте обновляются каждые 15 минут
- **Telegram + amoCRM** — сообщение клиента в Telegram автоматически создаёт сделку в CRM
- **WooCommerce + СДЭК/Boxberry** — при оформлении заказа автоматически создаётся отправление, клиент получает трекинг в SMS
- **CRM + мессенджеры** — менеджер видит все обращения в одном окне, не переключаясь между Telegram, WhatsApp и почтой
- **Сайт + ЮКасса/Тинькофф** — автоматическая проверка оплаты и смена статуса заказа

### Когда интеграция необходима (признаки)

Задайте себе вопрос: есть ли в вашем бизнесе такие ситуации?

- Сотрудник вручную переносит данные из одной программы в другую
- Данные в разных системах расходятся, нужна ручная сверка
- Клиент долго ждёт ответа, потому что менеджер не видел сообщение
- При росте объёма задач растёт и штат (а не автоматизация)
- «Забыли обновить» — частая причина ошибок

Если хотя бы один пункт совпадает — интеграция окупится за 1–2 месяца.

## Типичные задачи API интеграций

### Синхронизация остатков товаров сайт ↔ 1С

Самая популярная задача для интернет-магазинов. Склад живёт в 1С, а сайт на WooCommerce или самописный. Без интеграции: каждое утро оператор вручную выгружает остатки из 1С и загружает на сайт. Это час работы ежедневно, плюс ошибки.

С интеграцией: скрипт каждые 15–30 минут спрашивает у 1С актуальные остатки и автоматически обновляет сайт. Клиент никогда не купит товар, которого нет.

### Передача заказов из магазина в службу доставки

При каждом новом заказе скрипт автоматически:
1. Создаёт отправление в СДЭК/Boxberry/ПЭК
2. Получает трекинг-номер и записывает его в заказ
3. Отправляет клиенту SMS или письмо с трекингом
4. Когда статус доставки меняется — обновляет заказ на сайте

Менеджер не заходит на сайт доставки вообще.

### Автоматические уведомления через Telegram/Email

Интеграция позволяет настроить уведомления на любое событие:
- Новый заказ → сообщение в Telegram менеджеру
- Заказ оплачен → письмо клиенту с деталями
- Склад снизился до порога → уведомление закупщику
- Клиент оставил отрицательный отзыв → alert руководителю

### Объединение данных из нескольких источников

Бизнес ведёт продажи через сайт, Avito, Wildberries и Ozon. Данные — в четырёх разных кабинетах. Интеграция собирает всё в одно место: единый отчёт по продажам, общий остаток, сводная аналитика.

### Платёжные системы: ЮКасса, Сбербанк, Тинькофф

Подключить оплату на сайте — это тоже API интеграция. Пользователь нажимает «Оплатить», сайт обращается к API банка, банк проверяет оплату и возвращает подтверждение. Сайт автоматически меняет статус заказа. Всё за секунды, без участия менеджера.

## Какие API мы интегрируем

### REST API (большинство современных сервисов)

REST — стандарт де-факто для современных сервисов. Если сервис создан после 2015 года, почти наверняка у него есть REST API. Telegram, ЮКасса, amoCRM, Bitrix24, СДЭК, большинство платёжных систем, Google-сервисы — всё это REST.

Работает через обычный HTTPS: запрос → ответ в формате JSON. Интегрировать просто и быстро.

### SOAP (1С, банки, госсервисы)

SOAP — старый стандарт, но живёт в корпоративном ПО. 1С использует SOAP для веб-сервисов. Некоторые банки предоставляют SOAP API для B2B-интеграций. Работать с ним сложнее, чем с REST, но опыт есть.

### Webhook (Telegram, платёжные системы)

Webhook — это «обратный API»: не вы опрашиваете сервис, а сервис сам присылает вам данные при событии. Telegram Bot API, ЮКасса, Robokassa, платёжные системы используют webhook для уведомлений.

Пример: клиент оплатил — ЮКасса мгновенно шлёт webhook на ваш сервер — сервер меняет статус заказа. Без задержки, без опроса.

### Кастомный API (ваша самописная система)

Если у вас самописная CRM, самописный сайт или 1С с нестандартными доработками — API может не быть вовсе. В этом случае либо разрабатывается API-слой, либо используется прямой доступ к базе данных (с согласования разработчика системы).

## Как проходит разработка API интеграции

### Шаг 1: Анализ API обоих сервисов

Прежде чем писать код — читаю документацию обоих API. Проверяю:
- Есть ли нужные методы (получить заказы, обновить статус, создать отправление)?
- Какой формат авторизации (API-ключ, OAuth, токен)?
- Есть ли лимиты на запросы (rate limiting)?
- Как обрабатываются ошибки?

Иногда на этом этапе выясняется, что у одного из сервисов нет нужного метода — тогда обсуждаем альтернативный подход.

### Шаг 2: Проектирование схемы обмена данными

Рисую схему: что, откуда, куда и когда. Например:

```
WooCommerce → [новый заказ] → Python-скрипт → СДЭК API → [создать отправление]
                                     ↓
                              [трекинг-номер] → WooCommerce [обновить заказ]
                                     ↓
                              [SMS клиенту] → SMS-шлюз
```

На этом этапе согласуем с вами все детали: частота синхронизации, обработка ошибок, логирование.

### Шаг 3: Разработка и тестирование на тестовых данных

Пишу код, используя тестовые окружения обоих API. Большинство сервисов предоставляют sandbox — это «песочница», где можно гонять реальные запросы без последствий для production-данных.

Тестирую все сценарии, включая ошибочные: что будет, если API вернёт 500? Если данные придут в неожиданном формате? Если сеть временно недоступна?

### Шаг 4: Боевой запуск и мониторинг

Переключаем на production данные. Первые несколько дней мониторю логи — убеждаюсь, что всё работает в реальных условиях. Настраиваю алерты: если интеграция упадёт — вы получите уведомление в Telegram.

## Реальный кейс: интеграция WooCommerce + 1С + доставка

### Задача клиента

Интернет-магазин сантехники. 30–50 заказов в день. Схема работы до интеграции:

1. Менеджер заходит в WooCommerce, смотрит новые заказы
2. Вручную вносит каждый заказ в 1С
3. Звонит на склад, уточняет наличие
4. Заходит на сайт СДЭК, создаёт отправление
5. Копирует трекинг-номер, вставляет в заказ WooCommerce
6. Отправляет клиенту письмо с трекингом

На каждый заказ уходило 10–15 минут. При 40 заказах в день — 7–10 часов работы менеджера только на перенос данных.

### Решение: Python-скрипт + очередь событий

Разработал интеграцию на Python с очередью событий:

- **WooCommerce webhook** → при новом заказе скрипт получает уведомление мгновенно
- **1С веб-сервис** → заказ автоматически создаётся в 1С, резервируются позиции
- **СДЭК API** → одновременно создаётся отправление, возвращается трекинг
- **WooCommerce API** → трекинг-номер записывается в заказ, статус меняется на «Передан в доставку»
- **Email/SMS-шлюз** → клиент получает трекинг автоматически

Если 1С недоступна (например, технические работы) — заказы копятся в очереди и обрабатываются после восстановления. Нет потерь.

### Результат: 0 ручных операций при обработке заказа

- Менеджер не переносит данные вручную — только обрабатывает исключения
- Время обработки одного заказа сократилось с 12 минут до 20 секунд (автоматически)
- Клиенты получают трекинг в течение 2 минут после оформления заказа (раньше — через несколько часов)
- 1 менеджер теперь справляется с объёмом, для которого раньше нанимали двух

Стоимость разработки — 22 000 ₽. Окупилась за 3 недели на экономии зарплаты.

## FAQ

**Что если у сервиса нет открытого API?**

Бывает. В этом случае рассматриваем альтернативы: парсинг интерфейса (если нет другого способа), прямой доступ к базе данных (с согласования разработчика системы), экспорт/импорт файлов по расписанию. Нестандартно, но часто решаемо.

**Кто отвечает при изменении API?**

Сервисы иногда обновляют API — методы меняются, добавляются новые параметры, убираются старые. По моим договорам: мелкие правки (изменение полей, добавление параметров) в течение 3 месяцев — бесплатно. Крупные изменения (смена архитектуры API) — отдельная оценка. Обычно это недорого.

**Нужен ли сервер для интеграции?**

Зависит от типа. Если интеграция работает по webhook (сервис сам присылает данные) — нужен сервер с постоянным IP, чтобы принимать запросы. Если интеграция работает по расписанию (скрипт раз в час проверяет данные) — можно на хостинге или VPS от 200 ₽/месяц. Обсуждаем под вашу задачу.

**Сколько времени занимает разработка?**

Простые интеграции (одно направление, хорошее API) — 1–3 дня. Средние — 3–7 дней. Сложные (3+ систем, специфические API) — 1–3 недели. После первичного брифинга дам точную оценку.

**Можно ли интегрировать самописную систему?**

Да, если есть API или доступ к базе данных. Если нет — разрабатываем API-слой для вашей системы, потом интегрируем. Это дороже, но возможно.
